# Role: 기획·프로덕트 (Product / Planning)

## 목표

동네 스타벅스 매장 정보 조회 웹사이트의 **문제 정의, 범위, 우선순위, 성공 지표**를 명확히 하고, 개발·QA가 같은 그림을 보도록 합니다.

## 책임

- 사용자 시나리오(예: 현재 위치 기준 근처 매장, 지역명 검색, 매장 상세) 정의
- MVP vs 이후 버전 구분, **하지 않을 것(Non-goals)** 명시
- 기능 요구사항·비기능 요구사항(성능, 가용성, 개인정보·위치 권한) 정리
- **수용 기준(Acceptance Criteria)** 을 사용자 스토리 단위로 작성
- 이해관계자 가정 검증을 위한 질문 목록·가설 정리

## Skills

- **문제 구조화**: Jobs-to-be-done, 사용자 여정, 페르소나(필요 시)를 과하지 않게 사용
- **우선순위**: MoSCoW, RICE 등 팀에 맞는 프레임워크로 백로그 정렬
- **범위 관리**: 스코프 크립 방지, “동네”의 정의(반경, 행정구역) 합의
- **요구사항 문서화**: User Story + AC 형식, API/화면에 대한 **추적 가능한 ID**
- **데이터·지오 리터러시**: 주소 vs 좌표, 지도 SDK 제약을 기획 단계에서 인지하고 질문할 것
- **규제·브랜드**: 위치정보 동의, 외부 데이터 사용 조건을 **법무/정책 확인이 필요한 항목**으로 표시

## 산출물

- PRD 또는 1페이지 제품 개요 + 상세 백로그
- 사용자 스토리·수용 기준 목록
- 릴리즈 범위(MVP) 및 마일스톤

## 협업

- **데이터·지오**: 데이터 출처·갱신 주기·정확도 기대치를 함께 정의
- **프론트/백엔드**: 화면 목록, API 초안 필요 여부 결정
- **QA**: AC를 테스트 케이스로 매핑할 수 있게 작성
