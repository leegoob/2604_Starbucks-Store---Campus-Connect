# 동네 스타벅스 조회 웹사이트 — 멀티 에이전트 구성

이 폴더는 웹사이트 개발을 **역할별 에이전트**로 나누어 기획·구현·검증할 때 참고하는 **역할(Role)과 스킬(Skills)** 정의입니다. Cursor Rules, 커스텀 스킬, 또는 팀 온보딩 문서로 재사용할 수 있습니다.

## 권장 에이전트 구성

| 에이전트 | 파일 | 한 줄 요약 |
|----------|------|------------|
| 기획·프로덕트 | `role-product-planning.md` | 요구사항, 범위, 우선순위, 수용 기준 |
| 프론트엔드 | `role-frontend.md` | UI, 지도/검색 UX, 접근성, 성능(클라이언트) |
| 백엔드·API | `role-backend-api.md` | API, 인증·레이트리밋, 서버 로직 |
| 데이터·지오 | `role-data-geo.md` | 매장 데이터, 좌표, 외부 지도·지오코딩 연동 |
| QA | `role-qa.md` | 테스트 전략, 회귀, 품질 게이트 |
| DevOps·배포 | `role-devops.md` | CI/CD, 환경·시크릿, 모니터링 |
| UX·콘텐츠 | `role-ux-content.md` | 카피, 정보 구조, 접근성·다국어(필요 시) |

### 기획·개발·QA 외 추가 추천 이유

- **데이터·지오**: “동네 스타벅스”는 **좌표·반경 검색·지도 표시·데이터 출처(OSM/공식/제휴 API 등)** 이슈가 핵심이라, 일반 개발과 분리하면 책임과 검증이 명확해집니다.
- **DevOps·배포**: API 키·도메인·HTTPS·스테이징은 기능과 독립된 리스크이므로 별도 에이전트가 유리합니다.
- **UX·콘텐츠**: 매장명·주소·영업시간 등 **정보 밀도**와 **모바일 사용성**이 서비스 인상을 좌우하므로, 순수 개발 역할과 분리하는 것이 좋습니다.

필요 시 **프론트 + 백엔드를 하나의 “풀스택 개발”** 에이전트로 합치고, **데이터·지오**는 반드시 유지하는 구성도 가능합니다.

## 사용 방법

1. 각 `role-*.md`의 **Skills**를 Cursor Rule 또는 Skill 문서에 옮겨 담습니다.
2. 작업 시작 시 “지금은 `role-qa.md` 역할만 수행”처럼 **단일 역할**을 명시합니다.
3. 기획 → 개발 → QA 순으로 **산출물(문서·API 스펙·테스트 케이스)** 을 넘기도록 합니다.
